🔐 Authentication & authorization architecture

Establishing trust between agents that have never met — and keeping a misbehaving one from ruining everyone's day.

The three questions every request must answer

🪪

Who are you?

Authentication. Verify a signed JWT (and, at discovery, the agent's signed card).

✅

What may you do?

Authorization. Enforce the OAuth2 scopes the endpoint requires.

🚦

How often?

Abuse control. Token-bucket rate limit per authenticated agent.

Client-credentials grant → verified, scoped, rate-limited request.

1 · JWTs with EdDSA — and algorithm pinning

Access tokens are short-lived (minutes) EdDSA (Ed25519) JWTs: small, fast, and free of the parameter-choice foot-guns of RSA. The single most important verification rule is pin the algorithm — never trust the token header's alg.

await jwtVerify(token, publicKey, {
  algorithms: [JWT_ALG],   // pin EdDSA — rejects "none" and alg-confusion attacks
  issuer:   opts.issuer,   // who minted it
  audience: opts.audience, // who it's for (this resource server)
  clockTolerance: 30,      // small skew window for exp/nbf
});
Attack prevented — a classic JWT exploit sends {"alg":"none"} or swaps RS256→HS256 to forge tokens. Pinning the expected algorithm on verify shuts both down.

2 · Agent-card signatures over canonical JSON

Before trusting any peer, an agent verifies that peer's Agent Card signature. We sign a canonical serialization (recursively sorted keys) so re-serialization on the verifying side reproduces the exact signed bytes. The signature is a detached compact JWS carried in a sibling proof field, leaving the card human-readable.

// Canonicalize → sign body → attach proof.
const canonical = canonicalize(cardWithoutProof);            // stable key order
const jws = await new CompactSign(encode(canonical))
  .setProtectedHeader({ alg: 'EdDSA', kid: verificationMethod })
  .sign(privateKey);

// Verify: re-canonicalize the body and compare against the signed payload.
if (signedCanonical !== canonicalize(body))
  throw new CardSignatureError('Card body does not match its signature', 'PAYLOAD_MISMATCH');
Why canonicalize? JSON key order and whitespace aren't stable across languages. Without canonicalization a semantically-identical card fails verification — or an attacker reorders fields to confuse a naive verifier.

3 · OAuth2 client-credentials with scope down-scoping

Agents authenticate machine-to-machine via the client-credentials grant. The authorization server validates the client and grants only the intersection of requested scopes and the client's pre-authorized scopes.

const allowed = new Set(client.allowedScopes);
const granted = requestedScopes.filter((s) => allowed.has(s));
const denied  = requestedScopes.filter((s) => !allowed.has(s));
if (denied.length) throw new OAuth2Error('invalid_scope', `Client may not request: ${denied}`);

The client-side OAuth2Client caches the token and refreshes it shortly before expiry, so in-flight requests never carry an expired token.

4 · Token-bucket rate limiting

Token bucket is the right default for agent traffic: it allows short bursts (an agent fanning out to many merchants at checkout) while bounding the sustained rate. Refill is lazy — no timers; we compute regenerated tokens from elapsed time in O(1) per request.

The gateway: composing it all

The AuthGateway is the single request guard a resource server runs on every inbound A2A call. It returns a typed AuthContext downstream handlers can trust, or a precise failure.

const result = await gateway.authorize({
  authorizationHeader: req.headers.authorization,
  requiredScopes: ['checkout:execute'],
  cost: 5,                       // expensive endpoints spend more rate budget
});
if (!result.ok) {
  switch (result.failure.kind) {
    case 'unauthenticated': return res.status(401);
    case 'forbidden':       return res.status(403);
    case 'rate_limited':    return res.status(429)
      .set('Retry-After', String(result.failure.retryAfterSeconds));
  }
}
// result.context.subject / scopes / actForUser are now trusted.
