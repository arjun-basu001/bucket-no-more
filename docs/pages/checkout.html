🧩 Multi-merchant checkout orchestration

How one basket commits across many merchants — atomically enough that the user is never half-charged.

The problem

A single Universal Cart spans N independent merchants, each with their own commerce backend you do not control. There is no global two-phase-commit coordinator across third parties, so true ACID atomicity is impossible. But the user's expectation is simple and absolute:

"Either everything I ordered is placed, or nothing is charged."

We satisfy that expectation with a Try-Confirm/Cancel (TCC) saga. The orchestrator is a distributed saga coordinator built on four adapter primitives and their compensations.

The adapter contract

The orchestrator never talks to a merchant directly. It speaks the MerchantAdapter interface, which models a compensatable, two-phase checkout. Every method must be idempotent with respect to its idempotency key — the single most important property for safe retries and rollback.

export interface MerchantAdapter {
  readonly merchantId: MerchantId;

  // Phase 1 — reversible
  reserve(cart, ctx):    Promise<Result<Reservation>>;   // hold inventory + lock price
  authorize(res, ctx):   Promise<Result<Authorization>>; // hold funds (no money moves)

  // Phase 2 — commit
  capture(auth, ctx):    Promise<Result<Settlement>>;    // move the held funds
  confirm(set, ctx):     Promise<Result<Settlement>>;    // book into fulfilment

  // Compensations
  cancelReservation(res, ctx): Promise<Result<void>>;
  voidAuthorization(auth, ctx): Promise<Result<void>>;
  refund(set, ctx):            Promise<Result<void>>;
}
Why split authorize from capture? We place a hold on funds and inventory across every merchant first, and only capture once the whole basket is guaranteed. If anything fails before capture, rollback is clean and free — no money ever moved. This mirrors the AP2 mandate model.

The two-phase flow

    participant O as Orchestrator
    participant A as Merchant A
    participant B as Merchant B
    Note over O,B: Phase 1 — reversible (ALL merchants, in parallel)
    O->>A: reserve
    A-->>O: Reservation (price + inventory hold)
    O->>A: authorize
    A-->>O: Authorization (funds hold)
    O->>B: reserve + authorize
    B-->>O: ok
    Note over O,B: Phase 2 — commit (only if ALL passed phase 1)
    O->>A: capture + confirm
    A-->>O: Settlement + confirmation
    O->>B: capture + confirm
    B-->>O: Settlement + confirmation
    O-->>O: status = COMMITTED
Happy path: hold everywhere, then commit everywhere.

The state machine

Each merchant branch is a small state machine. Illegal transitions throw at runtime, catching a whole class of orchestration bugs. The orchestration state is the saga log — persist it after every transition and a crashed coordinator can resume or roll back deterministically.

Rollback: the path that matters most

If any merchant fails phase 1, the orchestrator compensates every merchant that got as far as reserve/authorize. Nothing was captured, so the user is never charged.

Cheap rollback before capture; best-effort refund after.
// From orchestrator.ts — the top-level decision tree.
const phase1 = await this.runPhase1(state, request, correlationId, log);
if (!phase1.ok) {
  // Nothing captured yet → clean rollback for everyone.
  await this.compensateAll(state, request, correlationId, log);
  setOrchestrationStatus(state, OrchestrationStatus.PARTIALLY_FAILED_ROLLED_BACK);
  return ok(this.toOutcome(state));
}

const phase2 = await this.runPhase2(state, request, correlationId, log);
if (!phase2.ok) {
  // Past the cheap-rollback boundary → best-effort refunds, surfaced loudly.
  await this.refundCaptured(state, request, correlationId, log);
  setOrchestrationStatus(state, OrchestrationStatus.PARTIALLY_FAILED_ROLLED_BACK);
  return ok(this.toOutcome(state));
}

setOrchestrationStatus(state, OrchestrationStatus.COMMITTED);

Edge cases handled explicitly

Edge caseHandling
Partial phase-1 successFull rollback of every branch; user not charged
Reservation expiry between phasesChecked (with clock skew) before any capture; abort if expired
Partial phase-2 failureBest-effort refund of captured merchants + PARTIALLY_FAILED status
Transient network errorsBounded retries with full-jitter exponential backoff — only when retryable
Duplicate / retried callsPer-step idempotency keys: cartId:merchant:step
Compensation that itself failsLogged as an incident; never throws past the boundary
Empty merchant set / missing adapterRejected before any side effects

Idempotency keys

Retries are only safe if repeated calls are no-ops. The orchestrator derives a deterministic key per (orchestration, merchant, step), so a retried capture returns the same settlement instead of charging twice:

idempotencyKey: `${cart.cartId}:${merchantId}:${step}`  // e.g. "cart-001:merchant.roastery:capture"
